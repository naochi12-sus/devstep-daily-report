# チーム日報管理システム　要件定義書

## 1. 背景

- チームが目指す目標をチーム員が協力して達成するには、誰がどこまでタスクを進めているのか業務の可視化が必要である。
- チーム内で共有・コメントし合うことでコミュニケーションの活性化も期待される。

## 2. ターゲットユーザー

- 中小企業のチームリーダー・マネージャー
- リモートワークチームのメンバー

## 3. 解決する課題

- 日報を紙やExcelでの管理からWebでの一元管理へ移行する。
- 現在、個人で管理しているタスクをチーム内で可視化・共有し、業務の属人化を防ぐ。
- 初対面、ネットワーク越しのメンバー同士でもコミュニケーションの活性化を図る。
- 日報へのフィードバックが遅い課題を、コメント機能で迅速化する。

## 4. 目的・方針

- 上記3の課題を解決するためのチーム日報管理Webアプリケーションであり、誰もが簡単に扱えるアプリケーションとする。

## 5. 概要

- 企業・チームで使用する日報管理システム。メンバーが日々の業務内容を記録し、チーム内で共有・コメントし合うことで、業務の可視化とコミュニケーションの活性化を図る。

## 6. 必須機能一覧

### 6-1. ユーザー認証・管理

- メール/パスワードによる新規登録・ログイン・ログアウト
- パスワードリセット（メール送信）
- プロフィール編集（名前のみ。アバターは名前から自動生成）
- 未ログイン時は `/login` へリダイレクト

### 6-2. 日報作成（Create）

- タイトル（必須・50文字以内）
- 日付（必須・デフォルトは今日）
- カテゴリ（選択式：開発 / 会議 / その他 等）
- 内容（必須・2000文字以内）
- バリデーションエラーは画面に表示

### 6-3. 日報一覧表示（Read）

- 全チームメンバーの日報を表示
- 表示項目：タイトル・投稿者名・日付・カテゴリ・本文冒頭（50-100文字）・コメント数・作成日時
- デフォルトは作成日時の降順（新しい順）
- ページネーション（10-20件/ページ）

### 6-4. 日報詳細表示

- タイトル・投稿者名・アバター・日付・カテゴリ・本文全文・作成日時・更新日時・コメント一覧
- 編集・削除ボタン（自分の日報のみ表示）

### 6-5. 日報編集（Update）

- 自分が作成した日報のみ編集可能

### 6-6. 日報削除（Delete）

- 自分が作成した日報のみ削除可能
- 削除確認ダイアログを表示

### 6-7. 検索・フィルター機能

- 日付範囲検索（開始日・終了日を指定）
- ユーザー絞り込み（特定ユーザーの日報のみ表示）
- カテゴリ絞り込み
- キーワード検索（タイトル・本文から部分一致）

### 6-8. コメント機能

- 日報詳細画面からコメント投稿（500文字以内）
- コメント一覧表示（投稿者名・アバター・本文・投稿日時）
- コメント削除（自分が投稿したコメントのみ）

## 7. 実務要件

### 7-1. TypeScript 厳格運用

- `tsconfig.json` で `strict: true` を設定
- `any` 型の使用禁止
- 型定義ファイルの作成（`types/` ディレクトリ）

### 7-2. エラーハンドリング

- 全てのAPI呼び出しで例外処理を実装
- エラーメッセージをユーザーに表示
- エラー発生時にアプリが落ちないこと
- 未ログイン時のリダイレクト・セッション切れ対応
- 404エラー時の専用ページ表示

### 7-3. バリデーション

- フロントエンド：フォーム送信前のチェック・リアルタイムバリデーション・エラーメッセージ表示
- バックエンド：Supabase RLS による権限チェック・データベース制約

### 7-4. テストコード

- ユニットテスト（Vitest）：ユーティリティ関数・バリデーション関数等
- E2Eテスト（Playwright）：最低1シナリオ（例：ログイン → 日報作成 → 一覧表示）
- 目標カバレッジ：60%以上

### 7-5. CI/CD（GitHub Actions）

- PR作成時：TypeScriptビルド・Lintチェック・ユニットテスト実行
- `main` マージ時：Vercel 自動デプロイ

### 7-6. レスポンシブデザイン

- スマートフォン（375px〜）・タブレット（768px〜）・デスクトップ（1024px〜）に対応。
- 主要画面がスマホで使えること。

## 8. 実装しない機能（スコープ外）

- 画像やファイルのアップロード機能（ストレージ圧迫や複雑化を防ぐためテキストのみ）
- リアルタイムチャット機能（非同期のコメント機能に留める）
- いいね・リアクション機能（まずは基本のコミュニケーション機能に集中するため）
- 複数チームの作成・切り替え機能（1ユーザー1チーム固定とする）

## 9. ユーザーインターフェース

- デザイン方針はシンプルで実用的であること。
- Tailwind CSS を使用。

## 10. システム構成

### 10-1. フロントエンド

- Next.js 16 (App Router)
- TypeScript（バグを防ぐための厳格な型定義）
- Tailwind CSS + shadcn/ui
- PC・スマホで表示可能

### 10-2. バックエンド

- Supabase（データの保存）
- PostgreSQL（Supabaseの裏側で動いているデータベース）
- Supabase Auth（認証）
- RLS (Row Level Security)
    - `daily_reports`：全員が全件閲覧可能・自分の日報のみ作成・編集・削除可能
    - `comments`：全員が全件閲覧可能・自分のコメントのみ作成・削除可能

### 10-3. インフラ・CI/CD

- Vercel（ホスティング環境）
- GitHub（バージョン管理）
- GitHub Actions（CI/CD）

### 10-4. テストコード

- Vitest (Unit Test)
- Playwright (E2E Test)

### 10-5. その他

- DiceBear API (アバター自動生成)

## 11. データベース設計

### テーブル構成

| テーブル名      | 主なカラム                                                                          | 説明           |
| :-------------- | :---------------------------------------------------------------------------------- | :------------- |
| `users` (Auth)  | `id`, `email`, `name`, `avatar_url`, `created_at`                                   | ユーザー情報   |
| `daily_reports` | `id`, `user_id`, `title`, `date`, `category`, `content`, `created_at`, `updated_at` | 日報データ     |
| `comments`      | `id`, `report_id`, `user_id`, `content`, `created_at`                               | コメントデータ |

## 13. 工程計画

- 6週間で要件定義・設計・実装・テスト・CI/CD・仕上げまで一連の工程を完了させる。
- 実務レベルの開発フローを想定して期限も意識し、承認レベルのものとする。
- 実務要件をクリアし、ポートフォリオとして公開できる成果物を完成させる。
